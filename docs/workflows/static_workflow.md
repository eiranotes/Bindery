# 정적(AI 없이) 집필 워크플로우

Bindery는 AI가 꺼져 있어도(설정 → 오프라인 모드) 장기 연재 작업대로 성립해야 한다.
모든 단계 산출물이 사람이 직접 쓰는 Markdown/JSON 파일이기 때문이다.

## 흐름

1. **작품 생성** — 시작 화면에서 제목/작가를 넣으면 폴더 스켈레톤이 생성된다.
2. **소재** — `ideas/inbox/`에 직접 소재 파일을 쓴다 (frontmatter 형식은 기존 파일/`.keep.md` 참조).
   보드에서 씨앗/채택/보류/폐기로 옮긴다 (= 폴더 이동).
3. **세계관** — `characters/`, `world/…`, `relationships/`에 자산 파일을 직접 작성.
   `canon/setting-bible.md`를 직접 채운다 (바이블 조립의 오프라인 실행은 자산 목록 정리까지만 해준다).
4. **플롯** — 플롯 화면에서 "회별 계획 제안"(오프라인이면 빈 뼈대 행 생성) 후 표에서 직접
   제목/목적/beats/hook을 입력하고 회차별 승인. 진실은 `plot/plot-board.json`.
5. **회차 계획** — 브리프/장면 계획을 오프라인 생성하면 플롯 row 기반 뼈대가 파일로 생기고,
   파일 화면(또는 외부 에디터)에서 직접 채운다. `<!-- bindery:json -->` 블록을 함께 고치면
   앱이 구조를 다시 읽는다 (마크다운만 고쳐도 집필에는 지장 없음).
6. **집필** — 회차 화면 「후보·적용」 탭의 "현재 원고 직접 수정" 또는 파일 화면에서
   `story/chapters/{ep}/manuscript.md`를 직접 쓴다.
7. **마감** — 요약(오프라인이면 통계 요약)을 만들고 직접 보강 → 픽스.
   픽스가 스냅샷을 만들고 `status/resume-state.md`를 재생성한다.
8. **다음 회차** — 홈 화면의 재개 상태에서 이어간다.

## 오프라인에서 되지 않는 것 (의도된 제한)

- 수정 후보 생성 — 원고 훼손 방지를 위해 뼈대 대체를 하지 않고 거부한다.
- 정사 변경 proposal — 근거 없는 설정 변경을 지어내지 않는다. (요약·바이블은 직접 수정)
- QA — "판정 아님" 안내만 남는다. 사람이 직접 검토한다.

## 외부 도구 병행

프로젝트는 평범한 폴더이므로 Obsidian/VS Code/Git과 그대로 병행할 수 있다.
앱은 열 때마다 파일에서 상태를 다시 읽는다 (`.bindery/`도 텍스트 JSON이다).
