# AI 작성 파이프라인 전체 흐름

Updated: 2026-07-03

이 문서는 현재 Bindery의 `AI 작업` 화면에서 사용자가 어떤 버튼을 누르고, 그때 내부에서 어떤 단계가 실행되며, 결과가 어디에 남고, 원고에는 어떻게 반영되는지를 설명한다.

## 현재 판정

결론: **앱 흐름은 돌아간다. 실제 AI 품질은 로컬 CLI 연결 상태에 달려 있다.**

- `AI 작업` 화면은 `01 연결 -> 02 바이블 -> 03 실행 -> 04 검토` 흐름으로 구성되어 있다.
- `03 실행`에는 7단계 파이프라인이 있다: `컨텍스트 -> 초안 후보 -> 표현 분석 -> QA -> 수정 계획 -> 요약 -> 기록`.
- 각 단계는 완료되면 회차별 산출물을 만들고, 산출물은 화면의 `산출물 보관함`과 프로젝트 파일 `.bindery/artifacts/{episode}/{step}.md`에 남는다.
- 초안/수정 후보는 원고를 직접 덮어쓰지 않는다. 반드시 `04 검토`에서 diff를 보고 적용한다.
- 후보 적용 전에는 현재 에디터 버퍼 기준 스냅샷이 먼저 생성된다.
- 로컬 Codex CLI 어댑터 스모크는 통과했다: `bash scripts/verify_ai_pipeline.sh` -> `AI pipeline adapter smoke: PASS (codex)`.

주의할 점:

- 브라우저/데모 모드에서는 실제 AI가 아니라 mock 또는 로컬 폴백으로 흐름만 확인한다.
- Tauri 앱에서 CLI 연결이 성공하면 실제 에이전트 출력이 사용된다.
- CLI 실행이 실패하거나 필요한 형식이 없으면 단계별 폴백이 사용된다. 그래서 UI 흐름은 계속되지만, 결과 품질은 실제 AI 결과와 다를 수 있다.

## 큰 그림

```text
프로젝트 열기
-> 집필 화면에서 원고 작성/저장
-> AI 작업 01 연결에서 CLI 확인
-> AI 작업 02 바이블에서 설정집 확인 또는 생략
-> AI 작업 03 실행에서 단계별 산출물 생성
-> AI 작업 04 검토에서 후보 diff/QA/수정 계획 확인
-> 필요한 hunk만 원고에 적용
-> 저장/스냅샷/내보내기
```

핵심 원칙은 두 가지다.

1. AI는 원고를 바로 수정하지 않는다.
2. AI가 만든 결과는 후보, 보고서, 요약, 기록 같은 산출물로 남긴 뒤 사용자가 적용한다.

## 사용자가 보는 화면 흐름

### 1. 프로젝트를 연다

사용자 행동:

- 시작 화면에서 `새 작품`, `기존 폴더`, `샘플` 중 하나를 선택한다.
- 프로젝트가 열리면 보통 `story/chapters/ep001/manuscript.md`가 자동으로 선택된다.
- 왼쪽 바인더의 `회차` 탭에서 회차를 고르거나 `+ 새 회차`를 만들 수 있다.

내부 동작:

- 프로젝트 루트가 `projectStore`에 저장된다.
- 파일 트리가 `fileTreeStore`에 적재된다.
- 선택한 원고가 `editorStore.content`에 올라온다.

결과:

- 사용자는 `집필` 화면에서 원고를 직접 편집한다.
- 이 화면에는 AI 실행 버튼이 없다. 슬래시 명령도 AI를 바로 실행하지 않고 `AI 작업` 화면으로 안내한다.

### 2. AI 작업으로 들어간다

사용자 행동:

- 상단 탭에서 `AI 작업`을 연다.
- 왼쪽 레일에서 현재 단계 상태를 본다.

화면 구조:

| 단계 | 사용자가 하는 일 | 목적 |
|---|---|---|
| `01 연결` | CLI 종류, 명령어, 출력 방식, 데모 모드 확인 | 실제 AI 실행 가능 여부 확인 |
| `02 바이블` | 설정집 감지, 템플릿 생성, 또는 생략 | 프롬프트에 넣을 작품 기준 정리 |
| `03 실행` | 7단계 파이프라인 실행 | 후보/분석/QA/요약/기록 생성 |
| `04 검토` | 후보 diff, QA, 수정 계획, 반복 분석 확인 | 원고에 반영할 부분 선택 |

## 01 연결

사용자 행동:

1. `실행기`를 선택한다.
   - `Codex CLI`
   - `Antigravity CLI`
   - `Gemini CLI`
   - `직접 설정`
2. `명령어`에 실행 파일 이름이나 절대 경로를 넣는다. 기본값은 `codex`다.
3. `출력 방식`을 선택한다.
   - `stdout`: 명령 실행 결과를 터미널 출력으로 받는다.
   - `file`: 에이전트에게 지정 파일에 결과를 쓰게 한 뒤 읽는다.
4. 필요하면 `데모 모드`를 켠다.
5. `연결 테스트`를 누른다.

내부 동작:

- `testAgentCli`가 Tauri 명령 `test_agent_cli`를 호출한다.
- 네이티브 앱에서는 지정 명령어에 `--version`을 붙여 실행한다.
- 데모/브라우저 모드에서는 mock 응답을 반환한다.

결과:

- 성공하면 레일에 `연결됨` 상태가 표시된다.
- 실패해도 앱 집필 기능은 계속 쓸 수 있다.
- 단, 실제 AI 단계는 실패하거나 폴백 결과로 대체될 수 있다.

## 02 바이블

사용자 행동:

1. `AI 작업 -> 02 바이블`로 이동한다.
2. 감지된 설정 문서가 있으면 목록에서 확인한다.
3. 없으면 둘 중 하나를 선택한다.
   - `설정집 템플릿 만들기`
   - `바이블 없이 진행`

내부 동작:

- 파일 트리에서 `bible`, `canon`, `설정`이 포함된 Markdown 파일을 찾는다.
- Codex 항목이 있으면 함께 보여준다.
- 템플릿 생성 시 `canon/setting-bible.md`를 만든다.
- `바이블 없이 진행`을 누르면 `pipelineStore.bibleSkipped = true`가 된다.

결과:

- 설정집이 있으면 이후 프롬프트의 기준 자료가 된다.
- 설정집이 없어도 파이프라인 실행은 가능하다.
- 다만 원고만 보고 생성/QA하므로 장기 연속성 품질은 낮아질 수 있다.

## 03 실행

`03 실행`은 실제 파이프라인 본체다. 사용자는 단계별 `프롬프트`를 먼저 볼 수 있고, `실행`을 하나씩 누르거나 `전체 실행`을 누를 수 있다.

실행 전 사용자가 조정할 수 있는 값:

| 항목 | 설명 |
|---|---|
| `분량` | 자동, 약 1,500자, 3,000자, 5,000자, 8,000자 |
| `창의성` | 보수적/균형/과감 계열 지시 |
| `문체 강도` | 문체 지침서가 있을 때 유연/균형/엄격 |
| `추가 지시` | 이번 회차에서 지켜야 할 자유 지시 |

이 값들은 `초안 후보`와 `수정 계획/후보` 쪽 프롬프트에 반영된다.

### 프롬프트 미리보기

사용자 행동:

- 각 단계의 `프롬프트` 버튼을 누른다.
- 내용을 확인하고 `이 프롬프트로 실행`을 누를 수 있다.

내부 동작:

- `assemblePrompt`가 단계별 템플릿, 원고 frontmatter, 관련 Codex 항목, 원고 앞부분을 합친다.
- `초안 후보`와 `수정 계획` 계열은 `buildGuidanceText`가 만든 작성 지침도 포함한다.

작성 지침에 들어갈 수 있는 것:

- 집필 파라미터
- 문체 지침서
- 문체 시스템 PromptCapsule
- 컨텍스트 팩
- 최근 QA 이슈
- 수정 계획
- 표현 분석 결과
- 회차 요약

## 7단계 실행 상세

### 1. 컨텍스트

사용자 행동:

- `컨텍스트` 단계의 `실행`을 누른다.

내부 동작:

- 현재 회차 번호를 원고 경로에서 추출한다.
- 이전 회차 요약 산출물이 있으면 가져온다.
- `plot/open-threads.md`가 있으면 열린 떡밥으로 읽는다.
- 현재 원고에 등장하는 Codex 항목을 추린다.
- 이 자료들을 합쳐 컨텍스트 팩을 만든다.

결과:

- 산출물: `.bindery/artifacts/{episode}/context.md`
- 이후 초안 후보 프롬프트의 `컨텍스트 팩`으로 자동 포함된다.

### 2. 초안 후보

사용자 행동:

1. 후보 종류를 고른다.
   - `초안`
   - `이어쓰기`
   - `다시쓰기`
2. `초안 후보` 단계의 `실행`을 누른다.

내부 동작:

- 현재 원고 전체가 base로 전달된다.
- `buildGuidanceText`가 문체 지침서와 기존 산출물을 모아 guidance를 만든다.
- `generateCandidate`가 Tauri 명령 `generate_candidate`를 호출한다.
- 네이티브 명령은 `agent_prompt`를 만든 뒤 설정된 CLI를 실행한다.
- Codex CLI는 `codex exec --skip-git-repo-check --ephemeral -s read-only -o <tempfile>` 방식으로 실행하고, 결과 파일을 읽는다.
- 다른 CLI는 `file` 모드 또는 `stdout` 모드로 실행한다.

성공 결과:

- 실제 에이전트 출력이 후보로 저장된다.
- 현재 구현상 실제 CLI 성공 시 보통 후보 1개가 생성된다.

폴백 결과:

- CLI 출력이 없거나 실패하면 네이티브 절차형 후보 A/B 2개가 만들어진다.
- 이 경우 source에 `native fallback`이 표시된다.

결과:

- 후보는 `candidateStore`에 저장된다.
- 산출물: `.bindery/artifacts/{episode}/draft.md`
- 화면은 자동으로 `04 검토`로 이동한다.
- 원고 파일은 아직 바뀌지 않는다.

### 3. 표현 분석

사용자 행동:

- `표현 분석` 단계의 `실행`을 누른다.

내부 동작:

- 현재 열린 원고 텍스트를 클라이언트에서 바로 분석한다.
- 반복어, 상투 반응, 대사 태그, 문장 끝 패턴 같은 항목을 검사한다.
- 결과는 `analysisStore.repetition`에 저장된다.

결과:

- 산출물: `.bindery/artifacts/{episode}/analyze.md`
- 에디터 본문에 반복 표현 하이라이트가 표시될 수 있다.
- `04 검토`의 반복 표현 패널에서 분포를 확인한다.

### 4. QA

사용자 행동:

- `QA` 단계의 `실행`을 누른다.

내부 동작:

1. `buildQAPrompt`가 QA 프롬프트를 만든다.
2. 프롬프트에는 현재 원고, 이전 회차 요약, 문체 지침서가 들어갈 수 있다.
3. `runAgentText`로 실제 CLI QA를 먼저 시도한다.
4. 에이전트 결과에 `bindery:qa-json` 블록이 있으면 그것을 우선 사용한다.
5. JSON 블록이 없거나 실행 실패 시 `run_qa` 폴백을 사용한다.
6. 문체 지침서의 금지어 검사는 클라이언트에서 항상 추가로 수행된다.

결과:

- `qaStore.report`에 게이트/이슈가 저장된다.
- 산출물: `.bindery/artifacts/{episode}/qa.md`
- QA 이슈는 에디터 lint/검토 패널에 연결된다.

중요:

- QA는 점수표가 목적이 아니라 수정 가능한 이슈를 만들기 위한 단계다.
- 실제 AI QA가 실패해도 기존 QA 파일이나 기본 보고서로 폴백된다.

### 5. 수정 계획

사용자 행동:

- `수정 계획` 단계의 `실행`을 누른다.

내부 동작:

- 현재 `qaStore.report.issues`를 바탕으로 수정 계획 프롬프트를 만든다.
- `runAgentText`로 실제 CLI에게 체크리스트 생성을 요청한다.
- 실패하면 `generate_revision_plan` 폴백을 사용한다.
- `parseRevisionPlan`이 Markdown 체크리스트를 구조화한다.

결과:

- `revisionStore.items`에 수정 작업 목록이 저장된다.
- 산출물: `.bindery/artifacts/{episode}/revise.md`
- `04 검토`에서 수정 계획 패널로 확인한다.

### 6. 요약

사용자 행동:

- `요약` 단계의 `실행`을 누른다.

내부 동작:

- 현재 원고를 회차 요약 프롬프트로 보낸다.
- 성공하면 에이전트 요약을 사용한다.
- 실패하면 원고 앞부분과 기본 통계로 로컬 요약을 만든다.
- 결과를 `canon/summaries/{episode}.md`에도 쓴다.

결과:

- 산출물: `.bindery/artifacts/{episode}/summarize.md`
- 파일: `canon/summaries/{episode}.md`
- 다음 회차의 컨텍스트/연속성 QA에 활용된다.

### 7. 기록

사용자 행동:

- `기록` 단계의 `실행`을 누른다.

내부 동작:

- 현재 열린 원고 버퍼를 대상으로 스냅샷을 만든다.
- 스냅샷 id, sha256, 대상 파일, 분량을 기록 저널로 만든다.

결과:

- 산출물: `.bindery/artifacts/{episode}/commit.md`
- 스냅샷: `.snapshots/{snapshot-id}/...`

## 04 검토

`04 검토`는 AI 결과를 원고에 반영하는 유일한 구간이다.

화면 구성:

| 패널 | 역할 |
|---|---|
| 후보 diff | 현재 원고와 후보를 비교 |
| QA Dashboard | 게이트, 점수, 이슈 확인 |
| Revision Panel | 수정 체크리스트 확인 |
| Repetition Panel | 반복 표현과 분포 확인 |

### 후보 적용 흐름

```text
후보 선택
-> 현재 원고와 diff 확인
-> 전체 적용 또는 hunk별 적용
-> 첫 적용 전에 현재 버퍼 스냅샷 생성
-> editorStore.content 갱신
-> 사용자가 저장
```

중요:

- 적용 전 스냅샷은 디스크의 마지막 저장본이 아니라 현재 에디터 버퍼를 기준으로 만들 수 있다.
- 후보 적용 후에도 저장은 별도다. 사용자는 `집필` 화면에서 저장하거나 `⌘S`를 누른다.
- AI 결과를 한 번에 믿고 덮는 방식이 아니라, diff 단위로 선택하는 방식이다.

## 산출물 저장 위치

파이프라인 결과는 두 군데에 남는다.

| 위치 | 역할 |
|---|---|
| `localStorage`의 `bindery-artifacts` | 앱 화면에서 빠르게 보여주는 최근 산출물 캐시 |
| `.bindery/artifacts/{episode}/{step}.md` | 프로젝트 폴더에 남는 파일 산출물 |

파일 산출물 예시:

```text
.bindery/artifacts/ep001/context.md
.bindery/artifacts/ep001/draft.md
.bindery/artifacts/ep001/analyze.md
.bindery/artifacts/ep001/qa.md
.bindery/artifacts/ep001/revise.md
.bindery/artifacts/ep001/summarize.md
.bindery/artifacts/ep001/commit.md
```

이 중 `context`, `analyze`, `qa`, `revise`, `summarize`는 이후 초안/수정 후보 프롬프트에 다시 들어갈 수 있다.

## 데모 모드와 실제 실행의 차이

| 상황 | 동작 |
|---|---|
| 브라우저 개발 모드 | Tauri 명령이 없으므로 mock 데이터 사용 |
| 데모 모드 ON | Tauri 앱 안에서도 mock/로컬 폴백 사용 |
| 데모 모드 OFF + CLI 연결 성공 | 실제 CLI가 후보/QA/요약/수정계획 생성 |
| CLI 실패 | 단계별 native/mock/local fallback 사용 |

따라서 “버튼을 누르면 흐름이 돌아가는가?”는 예다.  
“실제 AI가 좋은 원고를 생성하는가?”는 연결한 CLI, 계정 권한, 프롬프트, 프로젝트 자료 품질에 좌우된다.

## 현재 한계와 확인 포인트

1. 실제 CLI 성공 시 초안 후보는 보통 1개다. A/B 비교가 꼭 필요하면 에이전트에게 두 후보를 쓰게 하거나, 네이티브 명령에서 2회 호출하도록 확장해야 한다.
2. QA는 `bindery:qa-json` 블록이 있어야 구조화가 가장 잘 된다. 에이전트가 이 형식을 어기면 폴백 보고서가 사용된다.
3. `run_qa`와 `generate_revision_plan` 네이티브 폴백은 안전망이지 고품질 생성기가 아니다.
4. 요약은 실패해도 로컬 요약이 생기지만, 장면 흐름/인물 상태/떡밥 품질은 실제 AI 요약보다 낮다.
5. 산출물 파일 쓰기는 실패해도 UI 흐름을 막지 않는다. 배포 전에는 `.bindery/artifacts/` 파일이 실제로 남는지 확인해야 한다.
6. 패키지된 `.app` 내부 전체 클릭스루 자동화는 아직 남은 검증 항목이다.

## 사용자에게 권장하는 실제 사용 순서

1. `집필`에서 현재 회차 원고를 어느 정도 작성한다.
2. `자료`에서 주요 인물/설정/플롯을 정리한다.
3. `문체`에서 문체 지침서를 만들고 `집필 반영`을 켠다.
4. `AI 작업 -> 01 연결`에서 Codex CLI 연결 테스트를 통과시킨다.
5. `AI 작업 -> 02 바이블`에서 설정집을 확인한다.
6. `AI 작업 -> 03 실행`에서 `컨텍스트`를 먼저 실행한다.
7. `초안 후보` 또는 `표현 분석 -> QA -> 수정 계획`을 실행한다.
8. `04 검토`에서 후보 diff와 QA 이슈를 확인한다.
9. 필요한 hunk만 적용한다.
10. 원고를 저장한다.
11. `요약`과 `기록`을 실행해 다음 회차 준비와 스냅샷을 남긴다.
